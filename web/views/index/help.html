Domain proxy mode

Use case: small program development, public account development, product demo

Scenario:

  • You own the domain proxy.com and a public server with IP {{.ip}}
  • You have two intranet sites: 127.0.0.1:81 and 127.0.0.1:82
  • You want to access 127.0.0.1:81 via a.proxy.com and 127.0.0.1:82 via b.proxy.com

    • Steps:

    • Resolve *.proxy.com to the public server {{.ip}}
    • In the client management page, create a client and write down the verify key
    • Open the host management of that client and add two rules: 1) host: a.proxy.com, target: 127.0.0.1:81, 2) host: b.proxy.com, target: 127.0.0.1:82
    • Run on the intranet client 
      ./npc -server={{.ip}}:{{.p}} -vkey=<client verify key>
    • You can now visit a.proxy.com and b.proxy.com

    Note: the public IP ({{.ip}}) above is auto-detected. Adjust it if you are in a test environment. If you want to use HTTPS, set the HTTPS port to 443 in the configuration file and add the certificate paths.

    TCP tunnel mode

    Use case: SSH, remote desktop and other TCP scenarios

    Scenario: You want to connect to port 22 of intranet machine 10.1.50.101 by accessing port 8001 of public server {{.ip}} for SSH access

    Steps:

    • Create a client in the client management page and write down the verify key
    • Run on the intranet client 
      ./npc -server={{.ip}}:{{.p}} -vkey=<client verify key>
    • In the tunnel management of that client, add a TCP tunnel: listen port (8001), target IP and port (10.1.50.101:22), choose compression and save.
    • Accessing the public server {{.ip}}:8001 is equivalent to accessing 10.1.50.101:22, e.g. ssh -p 8001 root@{{.ip}}

    Note: the public IP ({{.ip}}) above is auto-detected. Adjust it if you are in a test environment. The intranet client is assumed to be running.

    UDP tunnel mode

    Use case: intranet DNS resolution and other UDP scenarios

    Scenario: there is an intranet DNS server (10.1.50.102:53) you want to use from outside the intranet, public server is {{.ip}}

    Steps:

    • Create a client in the client management page and write down the verify key
    • Run on the intranet client 
      ./npc -server={{.ip}}:{{.p}} -vkey=<client verify key>
    • In the tunnel management of that client, add a UDP tunnel: listen port (53), target IP and port (10.1.50.102:53), choose compression and save.
    • Set the local DNS to {{.ip}}, which is equivalent to using 10.1.50.102 as the DNS server

    Note: the public IP ({{.ip}}) above is auto-detected. Adjust it if you are in a test environment. The intranet client is assumed to be running.

    SOCKS5 proxy mode

    Use case: access intranet devices or resources from outside as if you were on a VPN

    Scenario: use port 8003 of public server {{.ip}} as a SOCKS5 proxy to reach any intranet device or resource

    Steps:

    • Create a client in the client management page and write down the verify key
    • Run on the intranet client 
      ./npc -server={{.ip}}:{{.p}} -vkey=<client verify key>
    • In the tunnel management of that client, add a SOCKS5 proxy: listen port (8003), an optional username and password (Proxifier supports it, some clients do not), choose compression and save.
    • On the external machine, configure a SOCKS5 proxy with IP {{.ip}} and port 8003, and you have full access to the intranet

    Note: the public IP ({{.ip}}) above is auto-detected. Adjust it if you are in a test environment. The intranet client is assumed to be running.
